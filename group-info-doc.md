# 项目体系总览

## 背景与目标

我维护了 10 个 AI 项目，每个项目对应一个飞书群 + 一个 Git 仓库。本文档是飞书文档「项目体系总览」的介绍部分源文件，也是我日常操作所有项目的主入口。

**核心目标：在飞书文档里一览全局。** 顶部放核心项目索引网站链接，接着是整体介绍和道法术器框架，然后是一张项目总览表格让我快速扫视，最后是各项目的结构化属性表格（定位、目录、链接、Skill、待办）。不需要在表格里堆图标和自动生成的定位——那些去网站看就行。有了定位校准后，自动生成的定位本来就没有意义，校准替代它。

**数据流：**

```
registry + 仓库扫描 → GROUP_INFO.md ← 飞书文档校准 → 群 Pin / 网站
```

飞书文档是用户操作的总入口，GROUP_INFO.md 是 Agent 上下文，网站是公开输出，Pin 是群内快捷入口。四者各司其职。

**设计原则：第一性原理出发，可读性优先。** 清晰、易懂、一目了然。

## 道 · 法 · 术 · 器

| 层级 | 含义 | 对应项目 |
|------|------|----------|
| **道** | 方向与本质 | Learn-X、Life-X |
| **法** | 认知框架 | 投资观、教育观、AI 观 |
| **术** | 方法执行 | Research-X、Read-X、Invest-X |
| **器** | 工具实现 | lark-channel-bridge、Skills、Index |

## 数据流

```
各仓库 GROUP_INFO.md（脚本自动生成，校准从飞书文档注入）
    │
    ├──→ 定时更新（每 3 天）
    │       ├── GROUP_INFO.md 更新（注入飞书文档校准）
    │       ├── 飞书文档同步（表格 + 各项目详情）
    │       ├── 群置顶（逐群发送）
    │       └── 核心项目索引网站（按 show_on_site 过滤）
    │
    └──→ 手动触发：update-all + sync-feishu-doc + pin + sync-site
```

执行顺序（定时/手动触发时）：

1. `update-all --apply`：扫描所有仓库 → 生成 GROUP_INFO.md（注入飞书文档的校准）
2. `sync-feishu-doc --apply`：同步注册表数据到飞书文档（保留用户填写的校准和展示列）
3. 逐群 `pin --apply`：发送置顶摘要
4. `sync-site --apply`：从飞书文档读取展示列，过滤后生成网站
